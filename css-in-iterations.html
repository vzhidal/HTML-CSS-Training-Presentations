Level 1

“Html/Css are for html coders only”

  • Рассмотрим на примере абстрактного разработчика.
  • Большинство разработчиков приходит из других технологий.
  • html/css важен. SPA красивый js код но плохой html/css

Inline Styles
  • Обычно всё начинается с инлайн стилей
  • Наш разработчик не знает как структурировать проект
  • не знает зачем было введено разделение html и css
  • история html/css

Results

  • Almost impossible to maintain.
  • Almost impossible to make themes.
  • Widespread duplication of code.
  • дубликация кода заботит его больше всего

Impulse

Results

  • There is no so many copy-paste.
  • Super hard to maintain.
  • Super hard to make themes.
  • Почти ничего не поменялось. Уменьшилось колличество дублируемого кода, но поддерживать и расширять код легче не стало.

Impulse

I can separate content and styles.

  • Первое, что осознаёт наш разработчик, это что есть необходимость разделения css и html.
  • Style и тэги стилизации это плохо.
  • Первостепенную важность имеет контент. Cтилизация сайта вторична.
  • Нужна возможность изменения стилизации без изменения контента.
  • Нужно разделить css и html

Level Up

  • Происходит level up
  • Осознание первопричины появления css (разделение контента и стилизации) переводит нашего разработчика на новый уровень
  • После перехода на новый уровень, больше никогда не будет использовать inline styles

Level 2

“Let’s see how html coders do that”

  • Идея возникающая на втором уровне: "У меня есть знакомый верстальщик, нужно посмотреть как делает он"

First steps of separation

  • Let’s use separate files for styles and html.
  • Let’s use ids (just because they are fast).
  • Благодаря возникшей идее разделения html и css наш разработчик решает использовать разные файлы.
  • Наш контент и стили разделены - значит нужно использовать какие-то селекторы для наших стилей. Большинство начинают с id. Потому что выборки по id быстрее. Потому что колличество id в кооде меньше и меньше шансов получить side effect.

Comments for defining structure


.container {
    margin: 10px;
    padding: 10px;
}
/****** HEADER *******/
.header {
    background: red;
}
/****** END HEADER *******/

/****** BODY *******/
.content {
    background: red;
}
/****** END BODY *******/

/****** FOOTER *******/
.footer {
    background: red;
}
/****** END FOOTER *******/
  • блоки коментариев.
  • можно добиться некоторого разделения и структуризации своих стилей.
  • можно колапсить в некоторых ide
  • это зло

Results

  • A bit easier to maintain (just because it’s easier to find some logical parts).
  • There is a big chance to place a style to a wrong group.
  • наш разработчик разделил одну большую кучу своего css на более маленькие кучки в которые продолжил сваливать всё подряд.
  • Основные проблемы:
    • можно легко перепутать блоки из-за размера файла
    • сложно понять, какие блоки существуют, проще положить вниз файла
    • переиспользование css стилей из разных блоков
    • система контроля версий. 1) может положить стили в неверный блок при автоматическом мерже 2) мержится приходится чаще

Impulse

  • Place logical parts of your styles in separate files.
  • Performance optimizations shouldn't affect project structure.
  • It’s possible to use build tools instead of developing in one file.
  • Разделение на файлы очень важно. Это становится проблемой для верстальщиков, не знающих инфраструктуры.

Deep nesting

(css coupled to dom)

.container div {
    color: red;
}
.container div .container ul {
    color: green;
}
.container div .container ul li a span {
    color: blue;
}
  • глубокая вложенность используется для увелечения специфичности.
  • пример с последовательным изменением требований.
  • лежит такой код в разных частях вашего css, из-зи чего поддерживать его становаится невозможно
  • нельзя стилизовать свой css опираясь только на специфичность селекторов
  • к чему приводит: !important или ещё хуже style !important
  • Развенчание мифа: !important !important

Results

  • Hard to maintain.
  • Hard to read and modify.
  • Hard to override.

One line styles


.container div {color: red;}
.container div .container ul {color: green;}
.container div .container ul li a span {color: blue;}
  • Попытка улучшить код за счёт визуального форматирования.

  • Поиск в 2-ух координатах

  • Есть необходимость в дополнительной структуризации и жёстком именовании

Level 3

“I’m not an html coder, but I’m a programmer. I can do better!”

Think about scalable and modular css

  • Scalable and Modular Architecture for CSS
  • Object Oriented CSS
  • Don’t Repeat Yourself CSS
  • Block, Element, Modifier

Think in modules

  • After getting a mockup try to split it to modules.
  • Make it possible to reuse your modules.
  • Write container independent CSS.
  • Use strong naming conventions.
  • Думайте в модулях, даже если сейчас ваш модуль не переиспользуется, он может быть переиспользован позже. Каждая логическая часть должна быть вынесена в отдельный модуль.
  • Пишите переиспользуемы модули с возможностью расширения. Н е вносите в сои модули специфику, которая позже не позволит переиспользовать ваш модуль
  • Контейнер независимы css. Рассказать про концепцию контейнера и контента. Не смешивать стили. Пример с float и margin
  • Строгое именование как единственное средство разделения на сегодняшний день. пример .text. Решает проблему Overspecifity

в сумме всё это позволяет вносить изменения в свой css на поздних этапах разработки без боязни всё разрушить.
